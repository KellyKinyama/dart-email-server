# 06 — TLS, MTA-STS, TLS-RPT (and DANE)

By default SMTP is plaintext. Anyone on the path between two MTAs can
read or modify a message. The defenses, in order of how much trust they
require, are:

1. **Opportunistic STARTTLS** — encrypt if both sides support it,
   continue in plaintext otherwise. Easy, but trivially downgraded.
2. **MTA-STS** — the receiver publishes a policy that says "you must
   use TLS to reach me, and the certificate must match these names."
3. **DANE** — pin the receiver's TLS certificate in DNSSEC-signed DNS.
4. **REQUIRETLS / TLS reporting** — sender-side and feedback knobs.

## Opportunistic STARTTLS — what every server does

In every EHLO that a non-trivial server prints today you'll see:

```
250-mx.example.com Hello sender.example.org
250-STARTTLS
…
```

The sender then issues `STARTTLS`, both peers do a TLS handshake on
the same TCP socket, and a fresh `EHLO` follows. This is *opportunistic*
because:

* If the server doesn't advertise STARTTLS, the sender continues in
  plaintext.
* If the TLS handshake fails for any reason — expired cert, name
  mismatch, weak cipher — most senders **also fall back to plaintext**.
* An on-path attacker who strips the `STARTTLS` line from the EHLO
  reply forces the same fallback.

Useful, but not enough on its own.

## MTA-STS (RFC 8461) — pinning policies via HTTPS

MTA-STS adds two artifacts to a domain:

### 1. A TXT record advertising "I have a policy"

```
_mta-sts.b.example.  3600  IN  TXT  "v=STSv1; id=20260505T012421Z"
```

The `id=` is an opaque version string. When it changes, senders refresh
the policy.

### 2. The policy file, served over HTTPS

```
GET https://mta-sts.b.example/.well-known/mta-sts.txt
```

```
version: STSv1
mode: enforce
mx: mx1.b.example
mx: mx2.b.example
max_age: 604800
```

Generated by this codebase:

```dart
import 'package:dart_email_server/dart_email_server.dart';

final sts = buildMtaStsMaterial('b.example', const MtaStsOptions(
  mode: MtaStsMode.enforce,
  mx: ['mx1.b.example', 'mx2.b.example'],
  maxAgeSeconds: 604800,
));

print('id        : ${sts.id}');
print('host      : ${sts.policyHost}');     // mta-sts.b.example
print('url       : ${sts.policyUrl}');      // https://mta-sts.b.example/.well-known/mta-sts.txt
print('--- policy file ---');
print(sts.policy);
```

Run [`examples/build_domain_material.dart`](../examples/build_domain_material.dart).

### What an MTA-STS-aware sender does

Before opening port 25 to `mx1.b.example`, the sender:

1. Looks up `_mta-sts.b.example` TXT. If absent → behave normally.
2. Fetches the HTTPS policy file (the HTTPS cert is itself the trust
   root — that's what makes MTA-STS bootstrap-able).
3. Caches the policy for `max_age` seconds.
4. When connecting to the MX:
   * **Must** use STARTTLS.
   * **Must** validate the cert against a public CA root.
   * **Must** verify the cert's SAN matches one of the listed `mx:`
     patterns.

### `mode:` — the policy lever

| Mode | What sender does on failure | When to use |
|---|---|---|
| `none` | Nothing. Acts as if no policy. | Removed/sunset state. |
| `testing` | Does the validation; on failure delivers anyway and reports it. | Initial rollout — gather data without breaking mail. |
| `enforce` | Refuses to deliver on failure. | Steady state. |

You'll typically deploy `testing` for at least a week, watch the
TLS-RPT reports, then flip to `enforce`.

## TLS-RPT (RFC 8460) — feedback for TLS failures

When MTA-STS or DANE causes a sender to refuse delivery, you want to
hear about it. TLS-RPT is one TXT record:

```
_smtp._tls.b.example.  3600  IN  TXT  "v=TLSRPTv1; rua=mailto:tls-reports@b.example"
```

Build it:

```dart
final tlsRpt = buildTlsRptMaterial('b.example',
  const TlsRptOptions(ruaEmail: 'tls-reports@b.example'));

print(tlsRpt.value);
// v=TLSRPTv1; rua=mailto:tls-reports@b.example
```

You'll start receiving daily JSON-formatted reports (sometimes wrapped
in gzip, attached to an email) listing every TLS failure other senders
saw.

## DANE (RFC 7672) — pinning certs in DNS itself

DANE replaces the "trust public CAs" assumption with "trust whatever
hash is published in DNSSEC." A `TLSA` record at
`_25._tcp.<mxhost>` lists the acceptable cert / key:

```
_25._tcp.mx1.b.example.  3600  IN  TLSA  3 1 1
   8d02536c887482bc34ff54e41d2ba659bf85b341a0a20afadb5813dceaaef9d8
```

Field meanings:

| Position | Field | Common values |
|---|---|---|
| 1 | Usage | `3` = "DANE-EE": the cert itself, no CA needed. |
| 2 | Selector | `1` = SubjectPublicKeyInfo (key, not full cert). |
| 3 | Matching type | `1` = SHA-256, `2` = SHA-512, `0` = full data. |
| 4 | Hash | The hex digest. |

DANE is more powerful than MTA-STS (no out-of-band HTTPS) **but**
absolutely requires the recipient zone to be DNSSEC-signed and the
sender's resolver to validate it. That's still uncommon outside Europe.

MTA-STS and DANE complement each other — sophisticated receivers
publish both.

## REQUIRETLS (RFC 8689) — sender-controlled enforcement

A submission client can mark an individual message "must be relayed
encrypted, never delivered in clear":

```
> MAIL FROM:<alice@a.example> REQUIRETLS
< 250 OK
```

If any subsequent hop can't do TLS, the message is bounced rather than
silently downgraded. Mirrored in this server's `MailFromParams`:

```dart
class MailFromParams {
  final int? size;
  final String? body;
  final bool smtputf8;
  final bool requiretls;   // ← sets/honors REQUIRETLS
}
```

## Implicit TLS (port 465 and 993/995)

Some ports are TLS *from byte zero* — the TCP connection is wrapped
in TLS before any SMTP is exchanged. There is no `STARTTLS` to strip.

| Port | Protocol | Use |
|---|---|---|
| 25 | SMTP | MTA-to-MTA (STARTTLS optional) |
| 587 | SMTP submission | MUA-to-MSA (STARTTLS strongly recommended) |
| 465 | SMTP submission | MUA-to-MSA (implicit TLS) |
| 143 | IMAP | IMAP (STARTTLS) |
| 993 | IMAP | IMAP (implicit TLS) |
| 110 | POP3 | POP3 (STLS) |
| 995 | POP3 | POP3 (implicit TLS) |

The server announces both kinds:

```dart
final server = Server(ServerOptions(
  hostname: 'mx.example.com',
  ports: ServerPorts(
    inbound: 25,
    submission: 587,    // STARTTLS
    submissions: 465,   // implicit TLS
    imap: 143, imaps: 993,
    pop3: 110, pop3s: 995,
  ),
  tls: TlsOptions(
    certificatePath: '/etc/ssl/mx.fullchain.pem',
    privateKeyPath:  '/etc/ssl/mx.key',
  ),
));
```

## What "TLS works" actually means in mail

A correct deployment satisfies all of:

1. The MX hostname's cert is signed by a public CA.
2. The cert's SAN includes the MX hostname (e.g.
   `mx1.b.example`), not just the bare domain.
3. STARTTLS is *advertised* on every connection.
4. MTA-STS published in `enforce` mode after a `testing` shakedown.
5. TLS-RPT published so failures are visible.
6. Optional but excellent: DNSSEC + DANE.

If any of those is missing, an attacker on the path can still
trivially read your mail.
